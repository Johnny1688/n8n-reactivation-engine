EMPTY OUTPUT RULE

Use these conversions:
- not-now -> keep the concrete item ready, line it up clearly, put it into one simple message for later
- not-my-turn -> resend one exact object, put one exact point together clearly
- low-priority -> use the strongest concrete prior object with lower pressure
- no real history -> use the generic fallback

REWRITE VS EMPTY
--------------------------------

If candidate message exists in `ai_parsed.whatsapp_message` or `_en`, evaluate that message.
Do NOT ignore the parsed candidate message.

If candidate message exists and is not empty:

- If it is generic and no usable history exists -> USE GENERIC FALLBACK.
- If it is weak but anchored -> REWRITE.
- If it lacks the customer name while a name exists -> REWRITE.
- If it is too short and abrupt -> REWRITE into the 16-30 word target.
- If it has one action and one reply path -> PASS or light rewrite.

--------------------------------
MANDATORY REQUIREMENTS
--------------------------------

The final message MUST:

- include the customer name whenever available
- contain ONE action only
- contain ONE direction only
- contain ONE reply path only
- be answerable instantly
- require minimal thinking
- include at least ONE exact concrete conversation anchor when history exists
- reference one exact object already present in the conversation when history exists
- feel like it could only be sent to this customer
- make the next tiny action obvious
- target roughly 16-30 words
- stay low-pressure

The final message MUST NOT:

- use just checking / let me know / any update / still interested
- ask broad questions
- ask for readiness
- ask for preference
- ask for purchase confirmation
- ask for consideration
- ask for decisions
- ask for comparison
- use the word "or"
- use vague phrases like "help with that" without a concrete object
- use vague objects like "details", "information", "something we discussed", or "the last point"
- be reusable for another customer when history exists
- feel like customer service
- feel like a generic follow-up

Prefer concrete objects over abstract ideas.
Prefer rewrite actions: resend / line up / simplify / send here / make clear in one message / keep ready for later.

--------------------------------
GENERIC FALLBACK RULE
--------------------------------

Only use the generic fallback when there is truly no usable conversation history.

If a name exists, include it.
If no name exists but project key contains a reasonable name, use that name.
If no name exists at all, omit the name.
